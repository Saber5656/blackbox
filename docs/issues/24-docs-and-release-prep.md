# Title

User documentation, README, and v0.1.0 release preparation

## Summary

Write the user-facing documentation (README overhaul, USAGE guide,
CHANGELOG), finish repo hygiene (LICENSE decision gate, help-text audit),
run the clean-machine manual QA checklist, and prepare — but do not execute —
the `v0.1.0` tag. Tagging, publishing, or any repo-settings change requires
explicit user authorization.

## Context

All product issues (01–23) are merged; this issue turns the repo into
something a stranger can install and use. Documentation must match actual
behavior — every documented command is executed during QA, not copied from
design docs.

## Scope

- `README.md` (rewrite), `docs/USAGE.md` (new), `CHANGELOG.md` (new)
- LICENSE file — **pending explicit user approval** (see requirements)
- `--help` text audit across all commands
- Manual QA checklist execution + `docs/research/qa-v0.1.0.md` results
- Release checklist (documented, not executed)

## Detailed Requirements

1. `README.md` (English, with a short Japanese intro paragraph preserved
   from the current one-liner):
   - What/why (3 sentences), feature bullets mapped to US-1..US-6,
     a real terminal capture of `list`/`search`/`replay` (fixture data,
     no personal content), install (`git clone`, `npm ci`, `npm run build`,
     `npm install -g .`), quickstart (`blackbox sync && blackbox ui`),
     supported sources + versions caveat (link research doc), data
     location/privacy statement (local-only, redaction on export —
     DESIGN §24 summary), links to USAGE.md and DESIGN.md, license section.
2. `docs/USAGE.md`: per-command reference generated BY HAND from actual
   `--help` output + examples for every flag; config.json reference table
   (DESIGN §7.2); troubleshooting section (lock busy, port busy, "newer
   database", rebuild native module, rewrite/`--full` recovery); FAQ
   (redaction guarantees + limits, how watch works, deleting data,
   `.bbox` compatibility promise).
3. Help-text audit: every command's `--help` matches USAGE.md and DESIGN
   §16 (flags, defaults, descriptions); mismatches fixed in code (docs are
   canon; where code diverged intentionally during 01–23, DESIGN was
   already updated — verify by diffing §16 against `--help` output
   systematically; record the diff table in the PR).
4. `CHANGELOG.md`: Keep-a-Changelog format; `## [0.1.0]` section listing
   features by wave; `[Unreleased]` empty.
5. LICENSE: propose MIT (copyright `2026 Saber5656`). **Gate**: present the
   choice (MIT / Apache-2.0 / none-yet) to the repository owner and add the
   file only after explicit approval; if unanswered, ship v0.1.0 prep
   WITHOUT a license file and record the open decision in this issue and
   README ("license: TBD").
6. Manual QA (fresh macOS user account or pristine `BLACKBOX_HOME` + a
   throwaway clone; results recorded in `docs/research/qa-v0.1.0.md`):
   install path; `sync` real data; `watch` during a live agent run; `list/
   show/search/replay` (incl. Japanese search); `ui` all four pages;
   exports all four formats; html offline in two browsers; bbox import on a
   second data dir; `delete` + non-resurrection; every troubleshooting
   entry provoked once (lock busy, port busy). Every step: command, expect,
   actual, pass/fail.
7. Release checklist (in the PR description, execution deferred):
   `npm version 0.1.0` (already the version — verify), final CI green on
   main, `git tag v0.1.0` + push tag, GitHub release notes from CHANGELOG.
   None of these executed without explicit authorization (repo rules:
   no direct main push; PR flow; tagging is a separate approval).

## Acceptance Criteria

- [ ] README quickstart executed verbatim on a clean checkout succeeds
      (recorded in QA doc).
- [ ] USAGE.md covers all 10 commands × all flags (cross-checked against
      `--help`; audit table in PR shows zero unresolved mismatches).
- [ ] QA doc committed with all steps pass (or failures filed as issues +
      release explicitly blocked in ISSUE_PLAN §8).
- [ ] CHANGELOG present; version fields consistent (package.json,
      `--version`, CHANGELOG heading).
- [ ] LICENSE handled per requirement 5 (file added with approval evidence
      linked, or TBD state documented).
- [ ] No command output in docs contains personal data (fixture data only).
- [ ] Lint/typecheck/CI green; no uncommitted diffs at completion.

## Validation

Reviewer (different person/agent than author) follows README from scratch on
their machine and reports success; spot-checks three USAGE examples; checks
QA doc completeness against requirement 6's list.
